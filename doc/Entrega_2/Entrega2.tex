\section{Introducción}

El presente documento recoge la fase de prototipado y diseño para el desarrollo de \textbf{Subsonic Festival}, una plataforma web \textit{full-stack} orientada a la gestión integral de festivales de música. Esta aplicación nace con el objetivo de centralizar y facilitar tareas críticas de la organización de eventos, tales como la promoción de artistas, la venta de entradas, la comercialización de \textit{merchandising} y la gestión del alquiler de espacios para proveedores de servicios.

El propósito principal de esta segunda entrega (Práctica 02) es diseñar las vistas para las funcionalidades descritas en la primera entrega, sin la necesidad de desarrollar las funcionalidades al uso. 

Para cumplir con estos objetivos, el contenido de este documento se organiza de la siguiente manera:

\begin{itemize}
    \item \textbf{En la Sección 2 (Esbozo de las vistas):} Se plantean los esbozos de las vistas, desarrollando los elementos que la componen y las ligaduras entre ellos. En cada vista, se presentará la funcionalidad que soporte, como se podrá acceder a ella, a que vista se navega desde ella y como.
    \item \textbf{En la Sección 3 (Árbol de navegación):} Se presenta un diagrama de estados que muestra el flujo lógico y global de navegación de la aplicación. En dicho esquema, se presentan diferentes eventos, como actuar ante ellos, además de como se tratarán los datos al realizar la navegación entre los eventos.
    \item \textbf{En la Sección 4 (Anexos):} Se adjuntan las actas de las reuniones de coordinación, evidenciando el seguimiento del proyecto.
\end{itemize}

En el \hyperref[fig1:arbol_nav]{árbol de navegación}, se muestra un diagrama de estados que representa el flujo lógico de navegación de la aplicación. A continuación, se detallan los distintos ejes de interacción.

El primero de los estados al que accedemos una vez cargado el sitio web es la página principal (\textbf{MAIN}), ya que se puede navegar por esta aún sin haber iniciado previamente sesión. 
\newpage

\subsection{Acceso y Autenticación de Usuario}

En caso de requerir una acción autenticada (necesaria para el uso del carrito de compras o gestión de entradas), el flujo se divide principalmente en dos ramas:
\begin{itemize}
    \item \textbf{Inicio de Sesión:} Si el usuario tiene una cuenta creada, bastará con introducir sus credenciales en el estado \textit{Iniciar sesión}. Tras la validación, será redirigido nuevamente al estado \textbf{MAIN}.
    \item \textbf{Registro de Cuenta:} En caso de no poseer cuenta, se transita al estado \textit{Añadir cuenta / Crear contraseña}. Una vez completados los campos y guardado el nuevo usuario, el sistema redirige al estado \textbf{MAIN} con la sesión ya activa.
\end{itemize}

% ---------- Descubrimiento y adquisición de eventos ----------
\subsection{Flujo de Descubrimiento y Adquisición de Eventos}
El usuario puede localizar los distintos eventos a través de dos vías:

\begin{itemize}
    \item \textbf{Exploración Pasiva:} Mediante el estado \textit{Recomendado}, que muestra contenido sugerido o recomendado (según valoraciones, fecha de proximidad, relevancia, etc).
    \item \textbf{Exploración Activa:} A través del estado \textit{Búsqueda}, donde se aplican filtros específicos que ayuden a cercar la búsqueda de eventos específicos.
\end{itemize}
Ambas vías convergen en el estado \textit{Detalles Evento}. Desde este punto, la transición \texttt{SELECT COMPRAR} redirige al usuario al módulo \textit{Métodos de Compra}, formalizando la transacción financiera.

% ---------- Flujo para cliente/usuario ----------
\subsection{Gestión de Perfil e Historial de Usuario}
Este eje está condicionado a la existencia de una sesión activa (\textit{Si está registrado}). El estado \textit{Perfil} actúa como nodo de control para:
\begin{itemize}
    \item \textbf{Configuración:} Acceso al estado \textit{Modificar Perfil} mediante la transición \texttt{SELECT AJUSTES}.
    \item \textbf{Consulta de Activos:} Transición al estado \textit{Historial}, donde el usuario puede visualizar sus entradas.
    \item \textbf{Gestión de Reembolsos:} Si una entrada cumple la condición \textit{Si entrada no consumida}, se habilita la transición \texttt{SELECT REEMBOLSAR}, que activa el estado de ejecución \textit{Action}.
\end{itemize}

% ---------- Flujo para proveedores ----------
\subsection{Búsqueda y Reserva de Espacios}
Para usuarios con rol de proveedor o aquellos interesados en logística, la aplicación ofrece un flujo dedicado a la infraestructura:
\begin{itemize}
    \item La transición \texttt{SELECT ESPACIOS DISPONIBLES} conduce al estado \textit{Búsqueda de Espacios}.
    \item Tras seleccionar un activo, se accede a \textit{Detalles del Espacio}, culminando el flujo en el estado \textit{Reservar Espacio}.
\end{itemize}

\section{Arquitectura de Aplicación}

En esta sección se presentará la arquitectura de la aplicación, detallando los componentes principales, sus responsabilidades y las interacciones entre ellos. Se describirá la estructura general del proyecto, incluyendo la organización de los archivos, las tecnologías utilizadas y cómo se integran. 

La arquitectura se ha basado en el patrón de diseño MVC + DAO + FACTORY, con el objetivo de mantener una separación clara entre la lógica de negocio, la presentación y el acceso a datos, facilitando así el mantenimiento y escalabilidad de la aplicación.

\subsection{Patrones de Diseño Utilizados}

\subsubsection{Modelo-Vista-Controlador (MVC)}
El patrón MVC divide la aplicación en tres capas principales:
\begin{itemize}
    \item \textbf{Modelo (Model):}  Representa las operaciones relacionadas con los datos y el modelo que maneja las entidades (como usuarios, eventos, entradas y espacios).
    \item \textbf{Vista (View):} Se encarga de capturar y presentar los datos del usuario. Es la interfaz gráfica que el usuario interactúa.
    \item \textbf{Controlador (Controller):} Intermediario entre el modelo y la vista. Maneja las solicitudes del usuario, procesa la entrada, interactúa con el modelo para obtener o modificar datos, y selecciona la vista apropiada para responder. Los controladores coordinan el flujo de la aplicación.
\end{itemize}
Este patrón facilita la separación de responsabilidades, permitiendo que cambios en la interfaz no afecten la lógica de negocio, y viceversa.

\subsubsection{Objeto de Acceso a Datos (DAO)}
El patrón DAO abstrae el acceso a la base de datos, proporcionando una interfaz uniforme para operaciones CRUD (Crear, Leer, Actualizar, Eliminar). Los DAOs encapsulan las consultas SQL y la lógica de persistencia, aislando el resto de la aplicación de los detalles específicos de la base de datos.
\begin{itemize}
    \item Beneficios: Facilita cambios en el sistema de almacenamiento (por ejemplo, de MySQL a PostgreSQL) sin modificar la lógica de negocio.
    \item En la aplicación: Cada entidad principal (usuario, evento, etc.) tendría su propio DAO, que implementaría métodos como \texttt{insert()}, \texttt{findById()}, \texttt{update()} y \texttt{delete()}.
\end{itemize}

\subsubsection{Fábrica (Factory)}
El patrón Factory se utiliza para crear objetos sin especificar la clase exacta que se va a instanciar. Proporciona una interfaz para crear familias de objetos relacionados, delegando la creación a subclases o métodos especializados.
\begin{itemize}
    \item Aplicación: En el contexto de DAO, una fábrica podría crear instancias de DAOs específicos según la configuración (por ejemplo, DAO para base de datos relacional vs. NoSQL).
    \item Ventajas: Reduce el acoplamiento, facilita la extensibilidad y permite la configuración dinámica de componentes.
\end{itemize}

\subsection{Estructura General del Proyecto}
La organización de archivos sigue una estructura modular que refleja los patrones MVC y DAO:
\begin{itemize}
    \item \textbf{Carpeta \texttt{model/}:} Contiene las clases del modelo que representan las entidades del negocio (por ejemplo, \texttt{Usuario}, \texttt{Evento}, \texttt{Entrada}). Incluye también los DAOs correspondientes para cada entidad.
    \item \textbf{Carpeta \texttt{view/}:} Alberga las plantillas y archivos estáticos para la presentación (HTML, CSS, JavaScript, imágenes).
    \item \textbf{Carpeta \texttt{conector/}:} Gestiona la conexión a la base de datos y posibles utilidades relacionadas con la persistencia.
    \item \textbf{Archivo \texttt{controller.py}:} Implementa los controladores que manejan las rutas y la lógica de control de la aplicación.
\end{itemize}

\subsection{Tecnologías Utilizadas}
La aplicación se desarrolla utilizando tecnologías web estándar para asegurar compatibilidad y escalabilidad:
\begin{itemize}
    \item \textbf{Backend:} Lenguaje de programación Python, con un framework web como Flask o Django para manejar las solicitudes HTTP y la integración con MVC.
    \item \textbf{Base de Datos:} Un sistema de gestión de bases de datos relacional como SQLite (para desarrollo) o PostgreSQL (para producción), accedido a través de una biblioteca ORM como SQLAlchemy.
    \item \textbf{Frontend:} HTML5, CSS3 y JavaScript para las vistas, posiblemente con una biblioteca como Bootstrap para estilos responsivos.
    \item \textbf{Integración:} Los patrones DAO y Factory se implementan utilizando conceptos de programación orientada a objetos en Python, como clases abstractas e interfaces.
\end{itemize}

\subsection{Interacciones entre Componentes}
\begin{itemize}
    \item El controlador recibe una solicitud HTTP y valida la entrada.
    \item Interactúa con el modelo a través de DAOs (creados por fábricas si es necesario) para realizar operaciones de negocio.
    \item El modelo procesa la lógica y devuelve datos al controlador.
    \item El controlador selecciona la vista apropiada y le pasa los datos necesarios.
    \item La vista renderiza la respuesta final para el usuario.
\end{itemize}
Esta arquitectura genérica permite que la aplicación sea mantenible (cambios localizados), escalable (componentes independientes) y extensible (fácil adición de nuevas funcionalidades).
